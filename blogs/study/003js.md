## 描述一下闭包

闭包（Closure）是指一个函数能够记住并访问其`词法作用域`（即定义时的作用域）的变量，即使这个函数在其作用域之外执行。简单来说，闭包 = `函数 + 该函数声明时所处的外部环境的引用`。

在 JavaScript 中，闭包通常通过在一个函数内部定义另一个函数，并将这个内部函数返回或传递给其他地方，从而让内部函数“记住”外部函数的变量。

##### 形成闭包的条件
1. 函数嵌套：存在外部函数和内部函数。
2. 内部函数引用外部函数的变量：内部函数使用了外部函数中声明的变量。
3. 外部函数将内部函数作为返回值返回（或者通过其他方式使内部函数在外部被调用），从而延长了变量的生命周期。

###### 优点
1. 数据私有化:闭包可以创建私有变量，避免全局命名冲突，同时提供对数据的受控访问。例如，只能通过特定的方法修改变量，而不能直接访问。
2. 延长变量生命周期:闭包中的变量不会被垃圾回收机制回收，可以在多次调用中保持状态，适用于计数器、防抖节流、函数柯里化等场景。
3. 模块化编程:利用闭包可以封装模块，只暴露公共接口，隐藏内部实现细节，有助于代码组织。

###### 缺点
1. 内存占用:闭包会一直持有外部函数的变量，导致这些变量无法被垃圾回收。如果滥用闭包，可能造成内存泄漏，尤其是在大型应用中。
2. 性能损耗:闭包涉及作用域链的查找，访问闭包中的变量比访问局部变量要慢一些。不过在现代引擎中，这种影响通常可以忽略，但大量闭包仍可能影响性能。
3. 可能导致意外的共享:在循环中创建闭包时，如果不正确处理，可能会共享同一个变量的最终值，导致不符合预期的结果（经典面试题）。

### 应用场景
1. 封装私有变量（模块模式），事件监听中保持上下文

```js
//私有变量 —— 模拟类的私有属性
function Person(name) {
  let _name = name; // 私有变量
  return {
    getName() {
      return _name;
    },
    setName(newName) {
      _name = newName;
    }
  };
}

const p = Person('Alice');
console.log(p.getName()); // Alice
p.setName('Bob');
console.log(p.getName()); // Bob
console.log(p._name); // undefined，无法直接访问私有变量
```

2. 防抖节流和函数柯里化与偏函数
   
```js
// 柯里化加法
function add(x) {
  return function(y) {
    return x + y;
  };
}

const add5 = add(5);
console.log(add5(3)); // 8
console.log(add5(10)); // 15

// 偏函数：固定部分参数
function fetchData(url, method, data) {
  // 模拟请求
  console.log(`请求 ${url}，方法 ${method}，数据：`, data);
}

const getJSON = fetchData.bind(null, '/api', 'GET'); // 用 bind 实现偏函数
getJSON({ id: 1 }); // 请求 /api，方法 GET，数据：{id:1}
```

3. 循环中保存异步操作的上下文:在循环中使用 var 或异步任务时，闭包可以“捕获”每次迭代的变量值，避免共享问题。

```js
// 经典问题：点击按钮输出对应索引
for (var i = 0; i < 5; i++) {
  (function(index) {
    document.getElementById(`btn${index}`).onclick = function() {
      console.log(`按钮 ${index} 被点击`);
    };
  })(i);
}

// 现代方法：直接用 let 声明块级作用域
for (let i = 0; i < 5; i++) {
  document.getElementById(`btn${i}`).onclick = function() {
    console.log(`按钮 ${i} 被点击`);
  };
}
```

4. 高阶函数与函数工厂:性能优化、单例模式、缓存计算结果

```js
// once：确保函数只执行一次
function once(fn) {
  let executed = false;
  return function(...args) {
    if (!executed) {
      executed = true;
      return fn.apply(this, args);
    }
  };
}

const init = once(() => console.log('初始化完成'));
init(); // 输出
init(); // 无输出

// memoize：缓存计算结果
function memoize(fn) {
  const cache = {};
  return function(arg) {
    if (cache[arg] === undefined) {
      cache[arg] = fn(arg);
    }
    return cache[arg];
  };
}

const fib = memoize(n => n <= 1 ? n : fib(n-1) + fib(n-2));
console.log(fib(40)); // 大幅提升性能
```

5. 迭代器与生成器

```js
function createRange(start, end) {
  let current = start;
  return {
    next() {
      if (current <= end) {
        return { value: current++, done: false };
      } else {
        return { done: true };
      }
    }
  };
}

const range = createRange(3, 5);
console.log(range.next().value); // 3
console.log(range.next().value); // 4
console.log(range.next().value); // 5
console.log(range.next().done);  // true
```

## 从输入 URL 到页面加载完成，发生了什么？

整个过程可以拆解为以下核心阶段：

#### 一、用户输入与导航启动

###### 判断输入类型

浏览器 UI 线程首先分析输入是**网址**还是**搜索关键词**。若是网址，自动补全协议（如 `https://`）；若是关键词，则调用默认搜索引擎拼接成搜索URL。

###### 安全检查与重定向

- 检查本地 **HSTS 预加载列表**，如果该域名强制 HTTPS，则内部将 `http://` 直接改写为 `https://`。
- 触发当前页面的 `beforeunload` 事件（可能弹出离开确认）。

###### Service Worker 拦截

如果该域已注册 Service Worker，则导航请求会被它拦截，可从本地 CacheStorage 提供离线内容，或代理请求。

###### 准备网络请求

浏览器主进程把“发起导航”任务交给网络线程（或网络进程），开始正式的资源获取。

#### 二、DNS 解析（域名 → IP 地址）

网络线程依次查询 DNS 缓存，顺序为：

1. 浏览器 DNS 缓存
2. 操作系统 hosts 文件
3. 本地路由器缓存
4. ISP（互联网服务提供商）的 DNS 服务器

若均未命中，ISP 的 DNS 服务器会发起**递归查询**：从根域名服务器 → 顶级域（.com/.org 等）服务器 → 权威 DNS 服务器，最终得到 IP 地址。  
现代浏览器会使用 **Happy Eyeballs** 算法同时尝试 IPv4 和 IPv6，并选用响应最快的。如果站点使用了 CDN，此时返回的 IP 将是离用户最近的边缘节点地址。

#### 三、建立网络连接

根据协议不同，连接建立过程有所区别：

###### 1. 经典 TCP + TLS（HTTP/1.1 或 HTTP/2）

- **TCP 三次握手**：`SYN` → `SYN-ACK` → `ACK`，在 TCP 层面建立全双工通道。
- **TLS 握手**（以 TLS 1.3 为例，更精简）：
  - 客户端发送 `ClientHello`（支持加密套件、密钥分享参数、SNI 等）。
  - 服务器回复 `ServerHello`，并附上**证书链**，随后用协商的密钥生成材料完成加密通道建立（1-RTT，若为 0-RTT 恢复会话更快）。
  - 浏览器验证证书：是否由受信 CA 签发、证书域名是否匹配、是否过期、吊销状态（OCSP 装订/CRL）等。
  - 对称会话密钥生成，后续应用数据都经过加密传输。

###### 2. QUIC（HTTP/3）

QUIC 基于 UDP，**将传输层加密和握手合并**。首次连接仅需 1-RTT 完成加密与流控建立，复用时甚至可实现 0-RTT，且无 TCP 队头阻塞问题，连接迁移更平滑。现代浏览器对支持 QUIC 的服务器会优先尝试。

---

#### 四、发送 HTTP 请求

连接建立后，浏览器构造 HTTP 请求报文：

- **请求行**：方法（GET/POST 等）、路径、协议版本。
- **请求头**：`Host`、`User-Agent`、`Accept`、`Cookie`、缓存协商字段（`If-None-Match` / `If-Modified-Since`）、安全头等。
- 若为 HTTP/2，会使用 **HPACK 头部压缩** 及**多路复用**在同一 TCP 连接上并发传输多个请求流；HTTP/3 则用 QPACK 压缩。

此时会先检查**浏览器 HTTP 缓存**：

- 强缓存命中（`Cache-Control: max-age` 未过期或 `Expires`） → 直接使用本地缓存，不发请求。
- 协商缓存 → 带 `ETag` 或 `Last-Modified` 条件发起请求，服务器返回 304 则用缓存。

#### 五、服务器处理请求

服务器端可能经历多层转发：

- **反向代理 / 负载均衡**（Nginx、HAProxy 等）接收请求，转发给具体应用服务器。
- 应用层处理业务逻辑，可能查询数据库、访问 Redis 缓存、调用内部微服务等。
- 若为**服务端渲染**（SSR），服务器会执行 JavaScript 生成完整 HTML 字符串。
- 最终组装 HTTP 响应：状态码（200、301、302、304、404、500 等）、响应头（`Content-Type`、`Cache-Control`、`Set-Cookie`、CORS 头、CSP 策略等）和响应体（HTML、JSON 等）。

---

#### 六、浏览器接收并处理响应

网络线程收到响应头后，立即解析：

###### 状态码处理

- `3xx` 重定向 → 根据 `Location` 发起新请求，并检查重定向次数防止死循环。
- `2xx` → 按 `Content-Type` 处理，`text/html` 则进入 HTML 渲染流程。
- `304` → 使用本地缓存资源。

###### 安全策略校验

- 检查 `Content-Security-Policy` 头，限制后续资源加载或脚本执行。
- 若 HTTPS 页面中混入 HTTP 资源（混合内容），浏览器可能拦截或降级处理。
- 应用 CORB/COOP/COEP 等跨域隔离策略。

###### 提交导航

当所有安全检查通过，网络进程把数据流交给渲染进程，浏览器 UI 开始更新地址栏、安全锁图标，并恢复访问历史。

---

#### 七、前端页面渲染（关键渲染路径）

渲染进程拿到 HTML 字节流后，展开密集工作：

###### 1. 构建 DOM 树

HTML 解析器（主线程）逐字节解码，通过**词法分析**生成 Token，再由语法分析构建 DOM 树。  
**预加载扫描器**（Preload Scanner）同时运行，提前发现 `<img>`、`<link>`、`<script>` 等外部资源，发出网络请求，减少等待。

###### 2. 加载与解析 CSS

遇到外部样式表 `<link>`，会发起请求并解析为 **CSSOM 树**（CSS 对象模型）。  
**CSS 不阻塞 DOM 解析，但阻塞渲染**，因为渲染树需联合 CSSOM。它也会**阻塞后续 JavaScript 执行**（JS 可能查询样式）。

###### 3. JavaScript 处理

- **同步脚本**（无 `async`/`defer`）：暂停 HTML 解析，等待脚本下载并执行完毕，因为脚本可能修改 DOM。
- **`defer` 脚本**：异步下载，在 DOM 解析完成后、`DOMContentLoaded` 事件前按顺序执行。
- **`async` 脚本**：异步下载，下载完立即执行，执行时可能中断 HTML 解析，执行顺序无关。

###### 4. 构建渲染树（Render Tree）

将 DOM 树与 CSSOM 树结合，生成只包含**可见节点**的渲染树（`display:none` 的元素不在其中，`visibility:hidden` 元素仍占位）。并计算出每个节点的**最终计算样式**（Computed Style）。

###### 5. 布局（Layout / Reflow）

从根节点开始遍历渲染树，计算每个元素的**几何信息**（坐标、宽高、盒模型边界等），生成布局树。可能因流式布局多次迭代。

###### 6. 分层与绘制（Paint）

浏览器对特定元素（如 3D 变换、`<video>`、`<canvas>`、`overflow:scroll` 等）创建独立的**合成层**，形成图层树（Layer Tree）。  
主线程为每个图层生成**绘制指令列表**（描述先画背景、再画边框等的顺序）。

###### 7. 合成与显示（Compositing）

- 绘制指令列表交给**合成线程**，它将每一图层切分成小块（Tile），交给 **GPU 进程** 进行栅格化（Rasterization，将矢量指令转为像素位图）。
- 所有图块完成栅格化后，合成线程通过 GPU 将它们拼接合成最终的屏幕图像，最后通过 IPC 交给浏览器主进程显示。

> **重要特性**：合成在合成线程完成，不占用主线程。这意味着即使主线程被 JS 阻塞，页面的滚动、CSS 动画依旧能流畅运行。


#### 八、页面生命周期与完成

- **`DOMContentLoaded` 事件**：DOM 构建完毕，所有 `defer` 脚本执行完，此时用户可看到部分内容，但图片等资源可能仍在加载。
- **`load` 事件**：页面及其所有子资源（图片、样式表、脚本、框架等）全部加载完毕。
- 之后浏览器进入可交互状态，并持续处理异步任务、更新 `requestAnimationFrame`、响应用户事件，形成完整的动态网页体验。


#### 九、贯穿全程的性能与安全优化

- **资源提示**：`<link rel="dns-prefetch">`、`preconnect`、`preload`、`prefetch` 在不同阶段提前做 DNS 解析、连接建立或资源缓存，大幅缩短后续请求的延迟。
- **缓存层次**：Service Worker Cache、HTTP 缓存、浏览器的内存缓存与磁盘缓存，让命中缓存的请求几乎零延迟。
- **安全防线**：混合内容拦截、CSP 策略、HTTP Strict-Transport-Security、CORS 校验等，在 URL 输入到渲染的每个环节都可能发挥作用。
